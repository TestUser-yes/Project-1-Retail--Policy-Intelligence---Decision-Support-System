#### 2. API Layer (app/api.py)

**Removed:**
- `import uuid` (line 4)
- UUID generation in AskRequest

**Before:**
```python
import uuid

class AskRequest(BaseModel):
    query: str = Field(..., min_length=3, max_length=10000)
    conversation_id: str = Field(default_factory=lambda: str(uuid.uuid4()))
```

**After:**
```python
class AskRequest(BaseModel):
    query: str = Field(..., min_length=3, max_length=10000)
    conversation_id: str = Field(default="", description="Optional conversation ID (auto-generated if empty)")
```

#### 3. Orchestrator (app/orchestrator.py)

**Removed:**
- `import uuid` (line 3)
- UUID generation: `query_id = str(uuid.uuid4())[:8]`
- query_id from logging and cost tracking

**Before:**
```python
import uuid

def run(self, query: str):
    query_id = str(uuid.uuid4())[:8]
    self.logger.log("input", {"query": query, "query_id": query_id})
    self.cost_tracker.record_query(query_id=query_id, ...)
```

**After:**
```python
def run(self, query: str):
    # query_id will be auto-generated by database (BigInteger autoincrement)
    self.logger.log("input", {"query": query})
    self.cost_tracker.record_query(...)
```

## Benefits

### 1. Storage Efficiency
- **Before**: String(36) = 36 bytes per ID
- **After**: BigInteger = 8 bytes per ID
- **Savings**: 78% reduction in ID storage per row

### 2. Performance
- ✅ Faster indexing (integer indexes are faster than string indexes)
- ✅ Faster joins (comparing integers vs strings)
- ✅ Smaller indexes (8 bytes vs 36 bytes)
- ✅ Less memory usage

### 3. Simplicity
- ✅ No UUID library dependency needed
- ✅ Automatic ID generation by database
- ✅ Human-readable IDs (e.g., user_id = 1, 2, 3...)
- ✅ Easier debugging

### 4. Database Native
- ✅ Uses database autoincrement feature
- ✅ Guaranteed uniqueness within table
- ✅ No application-side ID generation needed
- ✅ Works across all databases (PostgreSQL, MySQL, SQLite)
